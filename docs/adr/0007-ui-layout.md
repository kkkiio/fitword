# ADR 0007: UI Layout

## 状态

已采纳

## 背景

fitword 的 Web UI 需要承载对话交互和练习数据可视化。练习和写作不是两种模式——评分后 Agent 可顺势出题，两者在同一对话流中交替。唯一区分是输入框的发送意图不同。

## 决策

### 整体布局

```
┌──────────────────────────────────────────────┐
│  fitword（词感）                             │
├───────────┬──────────────────────────────────┤
│           │                                  │
│  💬 对话  │                                  │
│           │        主聊天区                  │
│  📊 统计  │                                  │
│           │                                  │
│           │                                  │
│           ├──────────────────────────────────┤
│           │  [提交评分]  输入框               │
└───────────┴──────────────────────────────────┘
```

- 左侧边栏：两个 tab，对话为默认
- 右侧主体：聊天区 + 底部输入框
- 输入框仅在聊天区下方

### Tab：对话

唯一的交互界面。Agent 行为：

- 默认以练习为主：在对话中主动出题（选择或填空）
- 用户可自然聊天、切换话题、指定题目格式
- 题目以卡片形式渲染在聊天区中

**输入框**：

- 底部固定，默认发送普通消息
- 点击「提交评分」按钮后，输入框变高、placeholder 变成"粘贴需要评分的文字..."，发送按钮文字变成"提交评分"
- 提交评分后 Agent 调 `evaluate_writing`，展示评分卡片
- 提交后按钮恢复默认状态

```
默认状态:           [提交评分]  ┌────────────────────┐  [发送]
                               │ 输入你的消息...     │
                               └────────────────────┘

评分状态:           [取消评分]  ┌────────────────────┐  [提交评分]
                               │ 粘贴需要评分的文字...│
                               └────────────────────┘
```

### Tab：统计

练习数据可视化，从 SQLite 读取。

- 总题数、正确率趋势、薄弱类型分布（柱状图）、选择题 vs 填空题对比
- 只读展示，v0.1 不做日期筛选和导出
- 切换到此 tab 时对话保持，返回对话 tab 时状态不变

### 题目卡片渲染

选择或填空题在聊天区中以卡片渲染，`____` 替换为交互组件：

- 选择题：4 个选项按钮，用户点击即提交
- 填空题：文本输入框 + 确认按钮
- 卡片背景与普通聊天消息区分，视觉上可识别为"题目"

### 评分卡片渲染

评分结果在聊天区中以卡片渲染：

- 总分 + 五维柱状
- 主要问题文字
- 可替换词列表（原词 → 替换词 + 理由）
- 改写版本全文

## 备选方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 两 tab（对话 + 统计） | 简洁，对话流统一 | — |
| 三 tab（练习 + 写作 + 统计） | 模式隔离 | 练习和写作实为同一对话流的不同操作，分离破坏连贯性 |
| 单页无 tab，统计融入对话 | 极简 | 数据可视化不如独立页面直观 |

## 后果

- 输入框需要评分开关的状态管理
- 题目卡片和评分卡片复用 pi SDK 的 tool result 渲染机制或自定义渲染
- 统计 tab 需要前端图表库（如 Recharts）
