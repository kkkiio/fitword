# ADR 0001: SQLite 本地存储

## 状态

已采纳

## 背景

fitword 需要持久化用户的练习数据（题目、答题记录、写作评分记录、统计信息）。原始 PRD 提出用 JSON 文件存储，但在后续讨论中确定需要支持统计查询（薄弱类型分布、合格率趋势等），JSON 文件的查询能力不足以支撑。

## 决策

使用 **SQLite** 作为本地存储引擎。

- 数据库文件位置：`~/.fitword/fitword.db`
- 仅管理业务数据（题目、答题记录、写作评分记录等；词语收藏夹后续如实现也归入业务数据）
- 对话历史由 pi SDK 的 jsonl session 机制管理，不入 SQLite
- 无需安装数据库服务，SQLite 为嵌入式单文件

## 备选方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| JSON 文件 | 零依赖，人类可读 | 查询需全量加载，统计困难 |
| SQLite | 标准 SQL 查询，统计方便，单文件 | 需要 SQLite 依赖 |
| PostgreSQL / MySQL | 功能强大 | 需要独立服务，违背本地优先 |

## 后果

- 统计数据查询变得简单，直接支持 Agent Memory 所需的各种聚合查询
- 引入 `better-sqlite3` 或类似 Node.js SQLite 库作为依赖
- 后续如需迁移到其他存储，SQL 标准降低迁移成本
- 对话历史与业务数据分离存储，需注意跨存储的关联查询
